# Agent notes for rtx

## `playground/` — the live-test harness directory

`playground/` (gitignored, not checked in) is where `rtx-mcp` and hand-driven mvdsv both run.
Required contents:

- `mvdsv` / `mvdsv.exe` — the QuakeWorld server binary.
- `id1/pak0.pak` and `id1/PAK1.PAK` — the base game data.

Everything else is optional / generated:

- `qw/maps/*.bsp` — test maps (`100m`, `aerowalk`, `bravado`, …), fetched separately.
- `qw/qwprogs.{dll,so,dylib}` — the game module. You do **not** need to stage this by hand:
  `server_start` / `server_restart` copy the newer of `target/release` and `target/debug` into
  place before launching (so a fresh `cargo build -p rtx-game` is picked up without a manual copy),
  unless called with `install_module=false` to reuse the already-staged module as-is. With nothing
  built and nothing staged, the call fails telling you to `cargo build --release -p rtx-game`
  first — it won't silently spawn a server that can't load its game logic. The returned status
  carries a `module` field naming which build went live.
- `qw/rjtest.cfg` — regenerated by every `server_start` / `server_restart`.

## Live bot testing — use the `rtx-mcp` MCP

Prefer the `rtx-mcp` MCP server (registered in the repo-root `.mcp.json`) over hand-driving
mvdsv when you need to run or observe bots live:

- `server_connect` attaches to a running server; `server_start` launches an isolated harness
  (see `playground/` above for what it needs, and its `install_module` flag).
- Configure with `set_cvars`, then `match_start` — it locks the roster and waits until the
  match is live (retrying only while the server stays in settled warmup).
- Verify a movement change with `corridor_test`: it reports drift, peak speed, and reverse
  frames for an A→B run (bots should hold 800+ ups on the 100m runway — see the corridor table
  below for known-good start/end points).
- Study play with `status` (match state, and each bot's team, stack, inventory, item goal,
  posture, enemy, route head, plus the oracle's plan and evaluation counters); `bot_route`
  expands a full route and `inspect_cell` explains the nav links around a cell.
- Rocket-jump work: `list_rj_links` / `test_links`; curl links: `list_curl_links`.

Link ids are not stable across `server_restart` or a `map` change — re-list after either. The
bridge lives in `crates/rtx-mcp/`; see its [README](crates/rtx-mcp/README.md).

### Known test-map corridors

`corridor_test` needs an A→B down a flat stretch of floor. Coordinates already mapped out (with
`inspect_cell` / the map's entity lump), so they don't need re-deriving:

- **`100m`** — `corridor_test(start=(224, -1408, 32), end=(224, 2900, 32))`, a straight run along
  +Y that holds 800+ ups. Keep the end short of `y≈2950`: the finish platform steps up near
  `y=3008`, and running into that ramp makes the bot overshoot and stall. Raise `max_cross_track`
  above the 64u default — a bhopping bot drifts ~100u off the centerline.

`aerowalk` and `bravado` aren't mapped yet; derive a corridor with `inspect_cell` around a spawn,
the same way.

## Build / test

- Default `cargo build` / `cargo test` cover only the game module, its nav core, and the wire
  codec. Build the MCP, viewer, and client explicitly: `-p rtx-mcp` / `-p navview` /
  `-p rtx-client`.
- Run `cargo fmt` before committing — the tree is rustfmt-clean, with `rustfmt.toml` pinning
  `max_width = 120`.
